## 2. 今日核心结论

> **智能体不是“接入了大模型的程序”，而是一个能够感知环境、基于目标做出决策、采取行动，并根据反馈继续调整的闭环系统。**

判断一个系统有多“Agentic”，关键不是它是否使用了 LLM，也不是它调用了多少工具，而是：

1. 它能感知多少环境信息；
2. 它能决定多少程序执行过程；
3. 它能否根据行动结果动态调整下一步；
4. 它是否受到明确的目标、权限和停止条件约束。

可以先用一句形象但不完整的话记忆：

> **模型像大脑，工具像身体，记忆保存状态，Agent Loop 让整个系统持续运转。**

## 3. 什么是智能体

在人工智能领域，智能体（Agent）是任何能够通过传感器感知环境，并自主地通过执行器采取行动，以达成特定目标的实体。

### 3.1 四个基本要素

| 要素 | 含义 | 旅行助手示例 |
| --- | --- | --- |
| 环境（Environment） | 智能体所处且能够被影响的外部世界 | 用户、天气服务、搜索服务、景点信息 |
| 传感器（Sensors） | 获取环境信息的渠道 | 用户输入、API 返回值、工具执行结果 |
| 执行器（Actuators） | 对环境采取行动的手段 | 调用天气 API、搜索景点、输出回答 |
| 自主性（Autonomy） | 根据感知、目标和内部状态独立决定行动 | 自主判断先查天气，再根据天气推荐景点 |

四者构成一个闭环：

```text
环境
  ↓ Observation
传感器 / 感知
  ↓
智能体内部决策
  ↓ Action
执行器 / 工具
  ↓
环境状态发生变化
  └──────────────→ 新的 Observation
```

### 3.2 “智能”的来源

仅仅能接收输入并返回输出，不足以说明它是一个完整智能体。真正重要的是自主性：系统能否根据当前状态选择下一步行动，并在结果不符合预期时调整计划。

因此，一个固定的问答接口更接近 LLM 应用；一个能够反复执行“观察、决策、行动、再观察”的系统，才更接近完整 Agent。

## 4. 传统智能体的演进路线

传统智能体不是一个单一类型，而是沿着“规则反应 → 状态建模 → 目标规划 → 效用权衡 → 从经验学习”的路径逐步发展。

| 类型 | 核心问题 | 主要能力 | 典型例子 | 局限 |
| --- | --- | --- | --- | --- |
| 简单反射智能体 | 当前发生了什么，我该立即做什么？ | 根据“条件-动作”规则响应当前输入 | 自动恒温器 | 没有记忆，无法理解上下文 |
| 基于模型的反射智能体 | 世界现在是什么状态？ | 用内部世界模型维护不可直接观察的信息 | 隧道中的自动驾驶汽车 | 能理解状态，但未必会规划目标 |
| 基于目标的智能体 | 怎样做才能到达目标？ | 搜索、规划并选择能导向目标的行动 | GPS 路径规划 | 只判断是否达成目标，难以权衡多个方案 |
| 基于效用的智能体 | 哪个结果最令人满意？ | 为候选结果赋予效用并进行权衡 | 同时考虑时间、油耗、拥堵的导航 | 效用函数设计困难 |
| 学习型智能体 | 如何从经验中变得更好？ | 根据反馈持续修正策略 | AlphaGo Zero | 需要大量数据、反馈与训练成本 |

这条演进路线的本质是：智能体对环境的理解越来越完整，对未来的考虑越来越深入，决策也从“能做”升级为“怎样做更好”。

## 5. LLM 驱动的智能体新范式

大语言模型没有改变智能体必须“感知—行动”的基本结构，但显著改变了智能体的决策核心。

### 5.1 传统智能体与 LLM 智能体的差异

| 对比维度 | 传统智能体 | LLM 驱动智能体 |
| --- | --- | --- |
| 决策核心 | 人工规则、搜索算法、效用函数 | 大语言模型的语言理解与推理能力 |
| 知识来源 | 工程师显式编码、知识库 | 大规模预训练知识、上下文和外部工具 |
| 输入形式 | 结构化、边界清晰 | 可直接处理自然语言和模糊目标 |
| 任务范围 | 通常面向单一专用任务 | 更通用，可处理跨领域任务 |
| 行为特征 | 稳定、确定、可预测 | 灵活，但可能产生幻觉和不稳定输出 |
| 开发重点 | 编写完整规则与流程 | 设计提示、工具、上下文、循环和约束 |

### 5.2 LLM Agent 的三项关键能力

以“规划一次厦门之旅”为例：

1. **规划与推理**：把模糊目标拆解成确认偏好、查询信息、制定行程、预订票务等子任务。
2. **工具使用**：识别信息缺口，主动调用天气、地图、搜索或预订 API。
3. **动态修正**：把“酒店超出预算”等用户反馈作为新约束，重新规划后续行动。

核心变化可以概括为：

> 开发者不再为所有情况写死完整流程，而是为模型提供目标、工具、状态和边界，让模型在约束范围内决定下一步。

## 6. 智能体的三种分类维度

同一个智能体可以同时属于多个类别，因为分类关注的角度不同。

### 6.1 按内部决策架构分类

- 简单反射型；
- 基于模型型；
- 基于目标型；
- 基于效用型；
- 学习型。

学习能力是一种“元能力”，可以叠加到前面多种架构上，而不只是完全独立的一类。

### 6.2 按决策时间与反应性分类

| 类型 | 特点 | 优点 | 缺点 | 示例 |
| --- | --- | --- | --- | --- |
| 反应式（Reactive） | 感知后立即行动，很少规划未来 | 延迟低、开销小 | 容易短视、陷入局部最优 | 安全气囊、高频交易 |
| 规划式（Deliberative） | 行动前评估未来状态与行动序列 | 有战略性，适合复杂任务 | 计算成本高，可能错过时机 | 下棋、旅行规划 |
| 混合式（Hybrid） | 结合快速反应与长期规划 | 兼顾灵活性和目标性 | 架构和协调更复杂 | 现代 LLM Agent |

现代 LLM Agent 的 Thought-Action-Observation 循环可以看作一种混合模式：Thought 负责审慎规划，Action 和 Observation 负责及时与环境交互。

### 6.3 按知识表示方式分类

| 范式 | 知识如何存在 | 优势 | 局限 |
| --- | --- | --- | --- |
| 符号主义 AI | 规则、逻辑、知识图谱等显式符号 | 透明、可解释、可追溯 | 规则维护成本高，难覆盖现实中的模糊与例外 |
| 亚符号主义 AI | 神经网络参数中的分布式统计模式 | 擅长图像、声音、语言等模式识别 | 黑箱、不擅长严格逻辑，可能产生幻觉 |
| 神经符号主义 AI | 神经网络与显式符号推理结合 | 兼顾感知、泛化、推理和可操作性 | 系统设计与协同更复杂 |

LLM Agent 可以视为神经符号主义的一种实践：LLM 是亚符号的神经网络核心，而计划、结构化工具调用、状态图和规则约束则属于可操作的符号部分。

## 7. 用 PEAS 描述任务环境

PEAS 用于在实现之前明确智能体的任务环境：

- **P — Performance**：怎样算完成得好？
- **E — Environment**：智能体在哪个外部世界中工作？
- **A — Actuators**：智能体可以采取哪些行动？
- **S — Sensors**：智能体可以获取哪些信息？

### 7.1 旅行助手的 PEAS 分析

| PEAS 要素 | 旅行助手中的内容 |
| --- | --- |
| Performance | 推荐相关性、信息准确性、预算符合度、响应速度、用户满意度、安全性 |
| Environment | 用户、城市、天气、景点、交通、酒店、票务平台、搜索服务 |
| Actuators | 调用 API、执行搜索、生成计划、提出澄清问题、向用户输出建议 |
| Sensors | 用户需求与反馈、天气数据、价格与余票、搜索结果、工具错误信息 |

### 7.2 LLM Agent 环境的典型特性

| 环境特性 | 含义 | 对设计的要求 |
| --- | --- | --- |
| 部分可观察 | 单次 API 调用无法获得全部真实状态 | 需要记忆、补充查询和信息来源标记 |
| 随机性 | 相同动作在不同时刻可能返回不同结果 | 需要处理不确定性、重试和降级 |
| 多智能体 | 其他用户、脚本或平台策略也在改变环境 | 需要并发意识、冲突处理和快速响应 |
| 序贯性 | 当前动作会影响后续选择 | 需要保存历史状态和行动轨迹 |
| 动态性 | 智能体思考时，价格、天气等仍可能变化 | 关键行动前需要重新确认实时状态 |

PEAS 的价值在于把“做一个智能旅行助手”这种模糊想法，变成可设计、可实现、可评估的工程问题。

## 8. 智能体的运行原理：Agent Loop

智能体不是一次性生成答案，而是通过持续循环逐步推进任务。

### 8.1 Agent Loop 的四个阶段

1. **感知（Perception）**：接收用户输入或上一步行动产生的 Observation。
2. **思考（Thought）**：理解状态、分解任务、更新计划、选择工具和参数。
3. **行动（Action）**：调用工具、执行代码、搜索信息或返回最终答案。
4. **观察（Observation）**：读取执行结果，并作为下一轮决策的新输入。

```text
Observation
    ↓
Perception
    ↓
Thought: 状态理解 + 任务规划 + 工具选择
    ↓
Action: 工具调用或 Finish
    ↓
环境执行并返回新的 Observation
    └────────────────────────────→ 下一轮
```

伪代码如下：

```python
history = [user_request]

for step in range(max_steps):
    decision = llm(history, system_prompt, tool_descriptions)
    action = parse(decision)

    if action.type == "finish":
        return action.answer

    result = execute_tool(action.name, action.arguments)
    observation = normalize(result)
    history.extend([decision, observation])

raise MaxStepsExceeded
```

### 8.2 Thought-Action-Observation 协议

课程中的第一个 Agent 使用了结构化文本协议：

```text
Thought: 分析当前状态并说明下一步
Action: tool_name(arg="value")
```

工具执行后，运行时把结果包装成：

```text
Observation: 工具返回的有效信息
```

如果任务完成，则输出：

```text
Action: Finish[最终答案]
```

这里有四个重要工程角色：

| 组件 | 职责 |
| --- | --- |
| System Prompt | 定义角色、目标、可用工具和输出协议 |
| Parser | 从模型输出中解析 Thought、工具名和参数 |
| Tool Registry / Executor | 校验并执行允许调用的工具 |
| History / State | 保存用户请求、模型决策和工具观察结果 |

Observation 不应只是把工具原始输出全部塞回模型。更稳妥的做法是先过滤冗余字段、保留来源和错误信息，再转换为模型容易使用的上下文。

## 9. 第一个 Agent：旅行助手

### 9.1 实验任务

用户请求：

> 你好，请帮我查询一下今天北京的天气，然后根据天气推荐一个合适的旅游景点。

这个任务至少包含两个有依赖关系的子任务：

```text
查询北京实时天气
        ↓
根据天气条件推荐景点
        ↓
整合结果并生成最终回答
```

第二步依赖第一步的 Observation，所以不能在缺少天气信息时直接可靠地完成。

### 9.2 实验组件

| 组件 | 本实验中的实现 |
| --- | --- |
| LLM | `deepseek-v4-flash` |
| 指令模板 | 规定 Thought、Action、Observation 和 Finish 格式 |
| 天气工具 | `get_weather(city)` |
| 景点工具 | `get_attraction(city, weather)` |
| 工具注册表 | 通过工具名查找并调用对应函数 |
| 主循环 | 最多执行若干轮，直到模型返回 `Finish[...]` |

课程实验还涉及 `requests`、`tavily-python` 和 `openai`：分别用于 HTTP 请求、实时搜索和兼容 OpenAI 接口的大模型调用。

### 9.3 我的实际运行记录

#### 循环 1：识别信息缺口并查询天气

```text
Thought: 用户请求查询北京天气并根据天气推荐景点，首先需要获取北京的实时天气信息。
Action: get_weather(city="北京")

Observation: 北京当前天气：Light Rain Shower, Thunderstorm，气温：26摄氏度
```

分析：模型没有直接凭训练知识猜测天气，而是识别出实时信息缺口，并选择天气工具。这体现了任务分解和工具选择能力。

#### 循环 2：利用上一轮结果推荐景点

```text
Thought: 获取到北京天气为雷阵雨、气温26摄氏度，接下来根据天气推荐景点。
Action: get_attraction(city="北京", weather="雷阵雨，26摄氏度")

Observation: Under a rainy, 26°C weather in Beijing, visit the Mutianyu Great Wall
for a scenic hike. The weather is warm, so wear light clothing. The wall offers
stunning views and a historical experience.
```

分析：模型正确地把第一轮的天气 Observation 转换成第二个工具的参数，说明历史记录已经参与下一轮决策。

这里也暴露出两个问题：

- 工具返回中英文混合，缺少统一的结果格式；
- 雷阵雨天气推荐长城徒步存在安全与体验风险，说明“工具有返回”不等于“推荐合理”。

#### 循环 3：停止循环并合成答案

```text
Thought: 已获取北京天气和景点推荐，信息足够，可给出最终答案。
Action: Finish[今天北京天气为雷阵雨，气温26摄氏度，推荐前往慕田峪长城游览，
雨中的长城别有一番景致，建议携带雨具并穿着轻便衣物。]
```

最终答案：

> 今天北京天气为雷阵雨，气温 26 摄氏度，推荐前往慕田峪长城游览，雨中的长城别有一番景致，建议携带雨具并穿着轻便衣物。

### 9.4 三轮循环体现的能力

| 能力 | 对应行为 |
| --- | --- |
| 任务分解 | 把“查天气并推荐景点”拆成两个顺序子任务 |
| 工具调用 | 自主选择 `get_weather` 和 `get_attraction` |
| 上下文理解 | 使用上一轮天气结果构造下一轮参数 |
| 结果合成 | 汇总天气与推荐信息，生成自然语言答案 |
| 终止判断 | 在信息足够后返回 `Finish`，不再继续调用工具 |

### 9.5 实验后的工程反思

这个 Demo 跑通了最小闭环，但距离可靠的生产级 Agent 还有明显距离：

1. **结构化输出不够稳**：用正则解析自由文本容易受到换行、引号和多余内容影响，后续可改用 JSON Schema 或模型原生 tool calling。
2. **参数缺少校验**：工具调用前应校验城市、天气、参数类型和必填字段。
3. **工具结果缺少标准契约**：建议统一返回 `status`、`data`、`error`、`source` 和 `timestamp`。
4. **推荐结果缺少验证**：雷阵雨时推荐长城不够合理，应加入安全规则、天气适配评分或二次验证。
5. **停止条件不能只交给模型**：除 `Finish` 外，还需要最大步数、超时、预算上限和重复动作检测。
6. **错误路径需要显式设计**：网络超时、工具不存在、返回为空时，应重试、换工具、降级或向用户说明限制。
7. **密钥必须隔离**：API Key 应通过环境变量或密钥管理服务加载，不能写入代码和提交到 Git。
8. **需要完整轨迹**：应记录每轮输入、模型输出、工具参数、工具结果、耗时与错误，便于调试和评估。

更合理的旅行推荐策略应先把天气转成约束：

```text
雷阵雨
  ↓
优先室内、交通方便、临时关闭风险低的景点
  ↓
检查营业时间、票务与安全提示
  ↓
给出主推荐 + 备选方案 + 风险说明
```

## 10. 我对 Agent 能力层次的理解

下面是我基于今天内容建立的个人能力分层。它不是教材中的正式分类，而是从“LLM 获得多少程序控制权”这一工程视角进行归纳。

> **回答 → 路由 → 调工具 → 自主循环 → 调度其他智能体**

| 层级 | 核心能力 | LLM 控制的内容 | 直观理解 |
| --- | --- | --- | --- |
| Level 0：回答 | 生成结果 | 只控制输出内容 | 简单 LLM 处理器 |
| Level 1：路由 | 选择流程分支 | 控制走哪条路径 | 判断任务属于哪一类 |
| Level 2：工具调用 | 选择工具与参数 | 控制具体动作 | 主动查询 API 或执行函数 |
| Level 3：自主循环 | 根据反馈决定下一步 | 控制任务推进过程 | 观察、行动、再观察 |
| Level 4：多 Agent 调度 | 分工、委派与汇总 | 控制多个智能体协作 | 像协调者一样组织专业 Agent |

控制权的变化可以表示为：

```text
只控制输出
    ↓
控制分支
    ↓
控制动作
    ↓
控制任务过程
    ↓
控制智能体协作
```

能力层次越高，系统不一定越好。自主性提升也会同步扩大成本、错误、权限和不可预测性的风险。工程设计应该选择“完成任务所需的最低自主性”，而不是盲目追求多轮循环或多 Agent。

---

## 11. Agent 的两种协作模式

### 11.1 作为开发者工具

Agent 深度嵌入人的工作流，负责补全、问答、重构、测试或调试。人仍然掌握目标、关键判断与最终验收。

典型产品包括 GitHub Copilot、Claude Code、Cursor 和 Trae。

这种模式的重点是“增强人”：自动化重复工作，让开发者集中精力处理架构、需求和关键决策。

### 11.2 作为自主协作者

人提供高层级目标，Agent 自主规划、执行、反思并交付结果。常见架构包括：

- 单 Agent 自主循环；
- 多 Agent 角色对话；
- 多 Agent 组织化工作流；
- 基于状态图的高级控制流。

相关框架或产品包括 AutoGPT、BabyAGI、CrewAI、AutoGen、MetaGPT 和 LangGraph。

从工具到协作者，交互关系发生了变化：

```text
命令 → 执行
```

变为：

```text
目标 → 委托 → 过程监督 → 结果验收
```

## 12. Workflow、Chatbot 与 Agent 的边界

| 维度 | Chatbot | Workflow | Agent |
| --- | --- | --- | --- |
| 核心目标 | 与用户对话并生成回复 | 按预设步骤稳定执行 | 自主选择行动以达成目标 |
| 流程控制者 | 应用代码或单轮模型 | 开发者预先定义 | 模型在约束内动态决定 |
| 是否使用工具 | 可选 | 可以使用 | 通常会使用 |
| 是否多轮 | 对话可以多轮 | 按固定流程推进 | 根据环境反馈动态循环 |
| 可预测性 | 中等 | 高 | 相对较低 |
| 适合任务 | 问答、客服、内容生成 | 规则稳定、合规要求高的流程 | 开放、复杂、路径难以预先穷举的任务 |

最简洁的区别是：

> **Workflow 预先规定“怎么做”，Agent 主要规定“要达到什么目标”，并允许模型决定部分过程。**

但两者不是非此即彼。生产系统通常更适合混合架构：

```text
Workflow 提供确定性的主干和安全边界
                  ↓
Agent 处理需要理解、推理和动态决策的局部节点
                  ↓
关键操作由规则校验或人工审批
```

例如退款系统可以用固定规则处理明确的小额申请，用 Agent 分析复杂例外，但最终仍由权限规则和人工审批控制高风险退款。

## 13. 如何判断一个系统是否应该做成 Agent

可以依次问五个问题：

1. 任务目标是否清晰，但实现路径难以提前完全写死？
2. 任务是否需要根据外部实时信息动态调整？
3. 是否存在多个工具，需要根据上下文选择？
4. 失败后是否需要观察原因、重新规划并继续尝试？
5. 使用 Agent 获得的灵活性，是否大于它增加的成本和风险？

如果流程稳定、规则明确、错误代价高，优先 Workflow；如果路径开放、信息不完整、需要推理与适应，Agent 更有价值。

## 14. 今日自测

### 14.1 为什么“会调用工具”不一定就是完整 Agent？

因为工具调用可能只是开发者写死的固定步骤。只有当模型能根据目标和观察结果选择工具、决定参数，并根据执行反馈调整后续行动时，系统才表现出更强的 Agent 特征。

### 14.2 为什么 Agent Loop 必须设置最大循环次数？

模型可能重复调用同一工具、无法解析错误、持续制定无效计划，形成无限循环。最大步数是运行时的硬停止条件，用于限制时间、Token、API 成本和外部副作用。

### 14.3 为什么只用最终答案准确率无法完整评估 Agent？

同样正确的答案可能来自完全不同的过程。还需要评估：

- 工具选择是否正确；
- 参数是否正确；
- 轨迹是否高效；
- 是否遵守权限和安全边界；
- 面对错误能否恢复；
- 成本、延迟和成功率是否可接受；
- 引用的信息是否真实、及时且可追溯。

### 14.4 本次旅行助手最值得优先修复的问题是什么？

不是更换更强模型，而是增加天气与景点的适配验证。因为当前轨迹虽然形式正确，却在雷阵雨条件下推荐户外长城，说明“执行成功”与“任务完成得好”不是一回事。

## 15. 今日总结

今天完成了从“把 Agent 理解为会调用工具的 LLM”到“把 Agent 理解为受目标和边界约束的环境交互闭环”的认知升级。

我目前对 Agent 的理解可以压缩成下面这句话：

> **Agent = 模型决策能力 + 可用工具 + 状态与记忆 + 运行循环 + 环境反馈 + 明确约束。**

今天跑通的旅行助手已经具备最小 Agent Loop：它能拆解任务、选择工具、读取 Observation、继续行动并主动结束。但它仍只是教学级 Demo。下一步不应该只增加更多工具，而应该先补上结构化协议、参数校验、错误恢复、停止条件、结果验证和轨迹评估。
